HIGH PRIORITY ISSUES
1. **Missing input truncation for Stripe metadata and product description**  
   `createBookingCheckout` passes `customerName` directly into `payment_intent_data.metadata` and `params.description` into a product description. Stripe enforces a 500‑character limit on metadata values and a 1000‑character limit on `description`. A long customer name or description will cause the checkout session creation to fail with a Stripe API error, blocking the booking.  
2. **No logging around Stripe API calls**  
   The module makes critical Stripe calls (`createBookingCheckout`, `captureDeposit`, `cancelDeposit`, `refundDeposit`, `constructWebhookEvent`) without any logging of requests or errors. Debugging payment failures requires external monitoring; sensitive errors may go undetected.

MEDIUM PRIORITY ISSUES
1. **Missing idempotency key in checkout session creation**  
   `createBookingCheckout` does not set an `idempotencyKey`. Retries of the same logical operation (e.g., network blips) could lead to duplicate holds if the creation route is retried without careful client‑side deduplication.  
2. **`refundDeposit` does not check PaymentIntent state**  
   Calling `refundDeposit` on an uncaptured PaymentIntent will throw a Stripe error. The function blindly attempts a refund; callers must guarantee the intent is captured, which is fragile.  
3. **`captureDeposit` / `cancelDeposit` / `refundDeposit` lack input validation**  
   A `paymentIntentId` of `undefined` or `null` would reach the Stripe API and throw an opaque error. Basic guard (`if (!paymentIntentId) throw …`) improves debuggability.  
4. **`applyDiscount` accepts any value for `percentOff`**  
   No check for non‑numeric or out‑of‑range values (e.g., negative, NaN). The `Math.max(0, …)` caps the final amount but does not prevent nonsensical inputs.

2. **Lazy `Stripe` client singleton is not thread‑safe**
