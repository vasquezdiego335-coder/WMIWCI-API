HIGH PRIORITY ISSUES
- **Potential database connection pooling hazard on serverless (Neon)**  
  The PrismaClient is instantiated with no explicit `connection_limit` or `datasource` configuration. When deployed on Vercel serverless functions against Neon (a connection-limited Postgres offering), the default Prisma pool size can quickly exhaust the database’s connection allowance, leading to `too many connections` errors and request failures. This is a well‑known Prisma‑serverless risk that requires client‑side tuning (for example, the `DATABASE_URL` must often include `?pgbouncer=true&connection_limit=1`). Because the architecture map states **Neon** and **Vercel**, the absence of this safeguard is a high‑impact reliability concern.

  The PrismaClient `log` array for `production` only contains `['error']`; warnings are suppressed. Prisma warnings (e.g., slow‑query thresholds, connection hiccups, or schema mismatches) could go unnoticed, delaying detection of subtle database problems. This reduces observability without meaningful cost.
